## 10.1 主线程的危机

Po 看着上一课留下来的难题：原生的 JavaScript 函数调用栈一旦开始递归，就无法被中断。如果一棵虚拟 DOM 树有 10,000 个节点，主线程就会被锁死好几百毫秒。

**🧙‍♂️**：还记得我们在第五章写的 `patch` 循环吗？

```javascript
// 曾经的旧代码：
for (let i = 0; i < children.length; i++) {
  patch(oldChild[i], newChild[i]);  // 深度优先的递归调用
}
```

在这里，我们依赖了 JavaScript 引擎的“调用栈”来记住我们遍历到了哪里。当你递归调用 `patch` 处理子节点时，引擎会在幕后帮你记住"等子节点处理完了，我要回到上一个父节点的循环里，继续处理下一个兄弟节点"。

**🐼**：是的。因为我们在依赖引擎的调用栈，所以我们没法跟引擎说“先停一下，我等会儿再回来”。既然我不能叫停 JavaScript 引擎，那唯一的办法就是——**我不让引擎帮我记了，我自己写代码来记住遍历的状态。**

**🧙‍♂️**：精确的洞察。但这不仅仅是“怎么记”的问题，我们需要从根本上重新构思整个渲染引擎的运作方式。在我们写任何代码之前，让我们先在脑海中建立起新引擎的全局蓝图。

## 10.2 大战略：时间切片与双阶段

**🧙‍♂️**：假设我们有了某种魔法，可以随时暂停和恢复渲染工作。现在我们需要决定的是：**我们应该在什么时候暂停？**

**🐼**：当然是当浏览器忙碌的时候！如果用户正在打字或者滑动屏幕，我们就应该停下来，把主线程让给浏览器去处理用户的操作，等浏览器空闲了我们再继续。

**🧙‍♂️**：很好。浏览器提供了一个 API 叫 `requestIdleCallback`。它会在浏览器每一帧处理完布局和绘制后，如果还有空闲时间，就来叫醒我们干活。这就叫做**时间切片（Time Slicing）**。

**🐼**：太棒了！那我们就把渲染一棵大树的任务切成一个个小任务，每次 `requestIdleCallback` 叫醒我们时，就处理几个节点，然后检查时间，没时间了就再把控制权交回去。

**🧙‍♂️**：听起来很完美。但是，请想象一下这个画面：假设一棵大树只渲染了一半，时间用光了，我们停了下来。此时，用户在屏幕上会看到什么？

**🐼**：呃……用户会看到一个只有上半部分的新页面，下半部分还是旧的。或者干脆是个残缺的页面！这绝对不行！

**🧙‍♂️**：没错。这就好比一个画家在画肖像画。如果他每画几笔就把帘子掀开给观众看，体验极其糟糕。正确的做法是在幕后画好全图，然后“唰”的一下揭开画布。

**🐼**：所以，我们需要把工作分成两半。前半部分在幕后悄悄做，随时可以被打断；后半部分一口气展现出来，绝不能被打断！

**🧙‍♂️**：总结得非常到位。这就构成了现代 React 架构的两个泾渭分明的阶段：
1. **Render Phase（渲染阶段）**：在内存中逐步构建树，收集所有的改动。这个阶段可以被打断。
2. **Commit Phase（提交阶段）**：一旦 Render 阶段完成，一口气将所有收集到的改动同步到真实的 DOM 上。这个阶段不可中断。

## 10.3 双缓冲：草稿与蓝图

**🐼**：既然 Render 阶段是在“幕后悄悄做”，那在这个漫长的过程中，页面上显示的是什么？如果此时发生了“更新”，我们要怎么处理？

**🧙‍♂️**：好问题。这就引出了我们的下一个核心概念。想象你是一位建筑师。你面前有一张 **完工蓝图**，它描述了当前大楼的模样，也就是用户此刻在屏幕上看到的东西。

现在客户打来电话说："帮我把三楼的窗户改大一些。"你会怎么做？

**🐼**：我不会直接在原图上改，那样改坏了或者画到一半被叫去开会，图纸就毁了。我会拿出一张 **全新的草稿纸**，参照原图，一层一层地画新蓝图。画到三楼的时候改大窗户，其他楼层直接照搬原图。

**🧙‍♂️**：完全正确。我们用两个全局变量来代表它们：
- `currentRoot`：**完工蓝图**（当前显示在屏幕上的树）。
- `wipRoot`（Work In Progress Root）：**草稿纸**（正在内存中构建的新树）。

```
完工蓝图（currentRoot）          草稿纸（wipRoot）
┌────────────┐                 ┌────────────┐
│  1楼: 没变  │  ──照搬──→      │  1楼: 同上  │
│  2楼: 没变  │  ──照搬──→      │  2楼: 同上  │
│  3楼: 改窗户 │  ──修改──→      │  3楼: 大窗户 │
└────────────┘                 └────────────┘
```

画完整张草稿后，你“唰”的一下把新蓝图替换掉旧蓝图（Commit Phase）。这就是计算机图形学中著名的 **双缓冲（Double Buffering）** 技术。

**🐼**：那么在画草稿（Render 阶段）的时候，我需要时时刻刻看一眼原图（`currentRoot`）来做对比，找出哪里变了。

**🧙‍♂️**：是的。为了方便对比，新草稿上的每一个节点，都会有一个指针叫 **`alternate`**（替身），指向原图上同一位置的旧节点。这就是新旧节点之间的桥梁。

## 10.4 挣脱束缚：Fiber 链表

**🧙‍♂️**：现在，我们有了“时间切片”的宏观节奏，也有了“双缓冲”的工作模式。最后一块拼图是——回到 10.1 的问题：**如何构建一个可以随时暂停的遍历数据结构？**

在之前的虚拟 DOM 中，我们使用了一个典型的“树（Tree）”结构：一个节点有一个 `children` 数组。这种结构天生适合递归，但不适合随时随地暂停。我们需要一种结构，它能明确地指出：当前在处理哪个节点，以及处理完这个节点后，**下一步**应该去哪。

**🐼**：那我们该怎么改造树呢？

**🧙‍♂️**：React 团队抛弃了作为单独遍历入口的 `children` 数组。他们给每个节点增加了三个关键的导航指针：

- **`child`**：指向它的 **第一个** 子节点。
- **`sibling`**：指向它的 **右边紧挨着** 的兄弟节点。
- **`return`**：指向它的 **父节点**（处理完了就“返回”给父亲）。

这个拥有三大导航指针的全新节点数据结构，就是 **Fiber**。想象一段这样的 HTML：

```html
<div>
  <h1>
    <p></p>
    <a></a>
  </h1>
  <h2></h2>
</div>
```

如果我们把它转换成 Fiber 结构，它不再是一棵树，而是一张通过指针连结的网络（单向链表）：

![figure 10.4](../../website/public/images/figure_10_4.png)

**🐼**：每个节点都通过线连在一起了。有了这三个指针，就算在某个瞬间暂停，也永远知道“下一步该去哪”。

**🧙‍♂️**：没错。如果是你，你会怎么设计这套“找路”的遍历规则？为了能尽早把最深处的节点结构确定下来，我们应该优先往深处走。

**🐼**：优先往深处走……那规则 1 应该是：如果有 `child`（大儿子），下一步就去找 `child`。

**🧙‍♂️**：是的。如果走到死胡同，没有子节点了呢？

**🐼**：那就横向走。规则 2：如果没有 `child`，就去找 `sibling`（右边的亲兄弟）。

**🧙‍♂️**：如果连 `sibling` 都没有——说明它是家里最小的孩子，它父亲的所有孩子都遍历完了。这时候该去哪？

**🐼**：既然父亲这边的活都干完了……那应该通过 `return` 找父亲的兄弟？也就是找“叔叔”！如果父亲也没有兄弟，就继续往上找爷爷的兄弟，直到回到根部。

**🧙‍♂️**：思路很清晰。这就是 Fiber 架构最核心的魔力。它把不可控的 **递归调用栈**，变成了可以随时操控的 **遍历链表循环**。就像是一个指针在迷宫里一步步地走，我们随时可以停下来去干别的事。

## 10.5 引擎的蓝图 (WorkLoop)

**🧙‍♂️**：最后，让我们用一段伪代码，把我们今天建立的全局模型（时间切片、双阶段、遍历游标）全部串联起来，这就是我们新引擎的“心脏跳动”机制——`workLoop`。

在这个循环里，除了代表整棵草稿树的 `wipRoot`，我们还需要一个全局变量 **`workInProgress`**，用来充当在迷宫里行走的“游标”，记录当前正在处理的具体节点。

```javascript
// ====== 引擎状态 ======
let wipRoot = null;         // 正在构建的草稿树（Render 阶段起点）
let workInProgress = null;  // 遍历游标，当前正在处理的 Fiber 节点

// ====== 引擎的心跳 ======
function workLoop(deadline) {
  let shouldYield = false;
  
  // 阶段 1：Render Phase (可中断)
  // 如果游标还没走到头，且浏览器还有空闲时间
  while (workInProgress && !shouldYield) {
    // 核心逻辑：执行一个节点的工作，并返回迷宫的下一个节点
    workInProgress = performUnitOfWork(workInProgress);
    
    // 检查浏览器是否快没时间了
    shouldYield = deadline.timeRemaining() < 1;
  }
  
  // 阶段 2：Commit Phase (不可中断)
  // 如果游标走完了（草稿画完了），且确实有一棵草稿树等待提交
  if (!workInProgress && wipRoot) {
    commitRoot(); // 一口气同步到真实 DOM，并交接给 currentRoot
  }

  // 请求浏览器在下一个空闲帧再次唤醒我们
  requestIdleCallback(workLoop);
}

// 启动引擎
requestIdleCallback(workLoop);
```

**🐼**：哇，这段代码完美地翻译了我们刚才讨论的所有概念！`while` 循环就是 Render 阶段，`shouldYield` 负责时间切片的让步。当 `while` 循环因为 `workInProgress` 变成 `null` 而自然结束时，就触发了同步的 Commit 阶段！

**🧙‍♂️**：正是如此。在这张宏伟的蓝图里，只剩下两个黑盒需要我们在下一章去填补：
1. `performUnitOfWork` 内部是如何处理单个节点的？它是如何一边比对 `alternate` 一边画出新草稿的？
2. `commitRoot` 内部是如何一口气把草稿更新到页面上的？
