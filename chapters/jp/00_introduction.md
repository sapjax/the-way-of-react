## この本について

この本では、あなた自身の手で **React をゼロから再発明** していきます。

React の API の使い方を学ぶのではなく、**React が発明されるまでの進化の道のりを追体験** するのです。最も原始的な `document.createElement` からスタートし、あらゆる「痛み」を身をもって経験しながら、その解決策をごく自然に導き出していきます。そして最終的に、それらの解決策こそが React の核心的な設計思想であることに気づくはずです。

各章は同じパターンで進みます：**まず「痛み」を感じ、次に解決策を導き出し、最後にそれを自分の手でコードに落とし込む**。読み終える頃には、約 **400行** の「mini-react」が手元に残ります。それは「雀は小さくとも五臓を具う」のごとく、Virtual DOM、Time Slicing、Fiber Reconciliation、一括コミット、そして核心的な Hooks をすべて含んでいます。

## 執筆スタイル：ソクラテス式対話

この本は完全に **対話形式** で展開されます。物語を動かすのは二人のキャラクターです：

- **師父 (🧙‍♂️)**：フロントエンド技術の進化の歴史に精通した賢者。丸暗記させるような答えは教えません。段階を追った問いかけと導きによって、ポー自身に答えを導き出させます。
- **ポー (🐼)**：React の経験はないが、基礎がしっかりしており、好奇心旺盛で鋭い質問を投げかける賢いパンダ。

この物語の形式は、二つの大きなインスピレーションに基づいています：

**ソクラテス式問答法**——知識を一方的に教え込むのではなく、質問を通じて学生に真理を発見させる手法です。師父は最初から「React は Virtual DOM を使っている」とは言いません。代わりにこう問いかけます。「もし状態が変わるたびに画面全体を書き換えたいけれど、実際の DOM ツリーをすべて壊して作り直したくないとしたら、どうする？」——するとポーは自分の直感に従って、一歩ずつ Virtual DOM の概念に辿り着くのです。

***『オペレーティングシステム：3つの容易な断片』(OSTEP)* の文体**——Remzi と Andrea Arpaci-Dusseau によるこの OS の教科書は、軽妙な対話と洗練されたコードを駆使して、極めて複雑な概念を親しみやすいものにしました。OSTEP は証明しました。「最も深遠な技術原理は、最も自然な言葉で語ることができる」ということを。この本も同様の読書体験を目指しています。畏まって「勉強」する必要はありません。落語や物語を楽しむように、ポーと師父の会話についていくだけでよいのです。

各章の終わりには、**数百行以内で完結する、そのまま実行可能な HTML ファイル** が手に入ります。ブラウザでダブルクリックするだけで開き、コードを修正して、学んだばかりのことをすぐに検証できます。npm も webpack も、頭の痛いビルドツールも一切不要です。純粋な JavaScript とブラウザだけで十分です。

## この本は誰のためのものか

この本は、JavaScript、HTML、DOM 操作の基本的な知識があることを前提としています。
React に触れたことがない、あるいはフロントエンドフレームワークを使ったことがなく、ゼロから React の明快なメンタルモデルを構築したい人。
あるいは、仕事で React を多用しているものの、その仕組みについてはあまり知らず、ブラックボックスの中身を覗いてみたい人。
どのような状況であれ、この本はあなたのためのものです。

私たちは React のメカニズムを一点ずつ手作りしていきます。すべての概念は対話の中で「第一原理」から導き出され、一歩ずつ確かな根拠に基づいています。
これは自己完結した旅です。JavaScript を知っていれば、旅の荷物は揃っています。さあ、出発しましょう。

## React を超えて

今日のフロントエンドの世界は、かつてないほど騒がしいものです。SolidJS、Svelte、Qwik、Angular Signals——新しいフレームワークやパラダイムが次々と現れます。

このような時代において、**React の API を丸暗記することは最悪の学習戦略です。** なぜなら、API は変わるものだからです。

**しかし、その背後に隠された「エンジニアリングのトレードオフ」は、決して簡単には変わりません：**

- 宣言型 (Declarative) vs 命令型 (Imperative)
- 全量再レンダリング (Full re-render) vs 細粒度更新 (Fine-grained updates)
- 実行時の柔軟性 (Runtime flexibility) vs コンパイル時の最適化 (React Compiler / Forget)
- Time Slicing 下の並行スケジューリング (Concurrent Mode) vs 従来の同期レンダリング
- クライアントサイド Hydration のコスト vs サーバーコンポーネント (RSC) のゼロ Bundle の理想

これらの次元におけるトレードオフと取捨選択を理解すれば、あなたが理解するのはもはや React だけではありません。**あらゆる UI フレームワークの設計意図** を見通すことができるようになります。将来どのような新しいフレームワークが登場しても、その底流にある核心的な思想を一瞬で見抜けるはずです。なぜなら、第一原理に至るまでの道すがら、あなたはすでにそれらを自分の手でこねくり回しているからです。
