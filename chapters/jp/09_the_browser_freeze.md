# 第九章：ブラウザの停滞 —— パフォーマンス危機の勃発 (The Browser Freeze)

![figure 9.1](../../website/public/images/ch09_browser_freeze.png)

## 9.1 スタックの底なし沼

**🐼**：師父、前章でロジックを再利用するために高階コンポーネントをたくさん追加しました。ところがテストしてみると、データが少し大きくなっただけで、入力時にページがカクつくようになってしまったんです。これはどういうことでしょうか？

**🧙‍♂️**：思い出してみろ。第五章で書いた `patch` 関数は、数千、数万のノードを含む仮想 DOM ツリーに遭遇したとき、どのように探索を行っていたか？

**🐼**：ルートノードを比較して、それから `for` ループで `children` を走査していました。もし子ノードもコンポーネントなら、 `patch` を再帰的に呼び出してさらに下へと進んでいきます。

**🧙‍♂️**：そうだ。JavaScript エンジン自身のコールスタック (Call Stack) に深く依存したこのアーキテクチャは、 **スタック調整器 (Stack Reconciler)** と呼ばれている。

**🐼**：再帰的なコールスタックに問題があるのですか？ ツリーを一気に比較し終えて、それから一括で実際の DOM を更新するというのは、理にかなっている気がしますが。

**🧙‍♂️**：コードのロジックとしては問題ない。だが、ブラウザの実行環境と組み合わせて考えるとどうだ？ フロントエンドにおいて、レンダリングとスクリプトを処理するスレッドはいくつある？

**🐼**：あっ！ フロントエンドはシングルスレッドでした。JavaScript の実行、ページのレイアウト、画面の描画、そしてユーザーのインタラクションイベントの処理……すべてが一本のメインスレッド (Main Thread) に詰め込まれています。

**🧙‍♂️**：その通りだ。では、アニメーションやページを滑らかに保つために、ブラウザは通常どれくらいのフレームレートを維持する必要がある？ それは、メインスレッドが一フレームに使える時間がどれくらいであることを意味する？

**🐼**：一般的には 60 FPS（1秒間に60フレーム）です。つまり、1000ミリ秒を60で割ると、一フレームのレンダリングに使える時間はわずか **16ミリ秒** ほどしかありません。

```text
|--- 16ms ---|--- 16ms ---|--- 16ms ---|--- 16ms ---|
  JS + 描画    JS + 描画    JS + 描画    JS + 描画
```

**🐼**：わかりました！ この 16ミリ秒の中で、ブラウザは JS を実行し、レイアウトを計算し、ピクセルを描画しなければならない。もし JS の実行に時間がかかりすぎると、描画のための時間が削られてしまうのですね。

## 9.2 傍若無人なメインスレッド占有

**🧙‍♂️**：そうだ。今度はその仕組みを我々の `patch` 関数に当てはめてみろ。もし 10,000 個のノードがあり、一回の再帰探索に 500ミリ秒かかるとしたら、何が起こる？

**🐼**：一旦 `patch` が始まると、JavaScript はコールスタックに何百何千という関数を猛烈な勢いで積み上げます。再帰は同期的ですから、この 500ミリ秒の間、メインスレッドは完全にロックされてしまいますね。

**🧙‍♂️**：その長い 0.5秒の間、ブラウザはどのような状態にある？

**🐼**：ブラウザがフリーズしたようになります！ 画面を再描画する暇もなければ、私のキーボード入力も処理できません。たとえ私がキーを押したとしても、その文字が表示されるのは 500ミリ秒の再帰が終わるまで待たなければなりません。これがカクつきの正体ですね。

```text
|------------- 500ms の純粋な JS 実行 (React Patch) --------------| -> | 描画 |
↑                                                          ↑
ユーザーのキー入力                                          画面がようやく更新
(ブラウザは応答不能)
```

**🧙‍♂️**：それだけではない。もしページ上で CSS アニメーションが動いていたり、ユーザーがページをスクロールしていたりすれば、それらはピタッと止まって見えるだろう。

## 9.3 制御権を譲ろうとする試み

**🐼**：あまりに自分勝手ですね。 500ミリ秒が長すぎるなら、細かく刻むことはできないでしょうか？ 例えば 10ミリ秒ずつの 50個の小さな塊に分けるとか。

**🧙‍♂️**：筋は通っている。一区切りつくたびに、ブラウザに入力処理や描画のための制御権を返し、暇ができたらまた続きを実行する。この技術を何と呼ぶ？

**🐼**：オペレーティングシステムの分野では「タイムスライシング (Time Slicing)」や協調的マルチタスクスケジューリング、といったところでしょうか。

**🧙‍♂️**：そうだ。だが、我々の `patch` コードを見てみろ。生の `for` ループと再帰の中で、そのような「中断」と「再開」を実現できると思うか？

```javascript
// Stack Reconciler の致命的な欠点：
function patch(oldVNode, newVNode) {
  // ... 他のロジック ...

  // 一旦ループに入り再帰が始まると、ツリー全体を走査し終わるまで、
  // 絶対に制御権を返すことはない！
  for (let i = 0; i < children.length; i++) {
    patch(oldChild[i], newChild[i]); // 深い再帰
  }
}
```

**🐼**：無理ですね……。生のコールスタックは滑り台のようなもので、一度滑り始めたら最後まで滑り落ちるしかありません。途中で止まったり、どこまで滑ったか覚えておいて後で再開したりすることなんて不可能です。

**🧙‍♂️**：その通りだ。JavaScript の生のコールスタックを使ってツリーを探索している限り、タイムスライシングは永遠に実現できん。

**🐼**：では、どうすればいいのでしょうか？ 自前でコールスタックをシミュレートして、ツリーの探索を再帰から「いつでも中断可能なループ」に変える必要があるのでしょうか？

**🧙‍♂️**：それこそがモダン React の核心的な転換点だ。この致命的なパフォーマンス危機を解決するために、React チームは丸二年の歳月をかけて、底レイヤーの調整エンジンを完全に書き換えたのだ。

**🐼**：完全に書き換えた？ 元々のツリー構造まで変えてしまうのですか？

**🧙‍♂️**：そうだ。彼らは生のコールスタックを捨て去り、「中断可能」で「タイムスライシング」に適した、全く新しい連結リスト構造を導入した。それが —— **Fiber** だ。だがその前に、今日の実験でメインスレッドがロックされる感覚を、まずはその身で味わってみるがいい。

### 📦 やってみよう

以下のコードを `ch09.html` として保存し、ブラウザで開いて、深い再帰によって引き起こされる「メインスレッドのロック」を体験してみよう：

```html
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>Chapter 9 — The Browser Freeze</title>
  <style>
    body { font-family: sans-serif; padding: 20px; text-align: center; }
    .box { display: inline-block; width: 10px; height: 10px; margin: 1px; background: #0066cc; }
    input { padding: 10px; font-size: 16px; width: 300px; margin: 20px; }
    button { padding: 10px 20px; font-size: 16px; cursor: pointer; }
    #status { color: #ff4444; font-weight: bold; margin: 10px; }
  </style>
</head>
<body>
  <h1>Browser Freeze Demo</h1>
  <p>1. 下の入力欄で文字を打ってみて、その滑らかさを確認してください。</p>
  <p>2. 「万単位のノードレンダリングをトリガー」ボタンをクリックします。</p>
  <p>3. <strong>レンダリングが完了する前</strong>（ページが固まっている間）、すぐに入力欄に戻って猛烈にタイピングしてください。</p>
  
  <input type="text" placeholder="ここでタイピングを試してください...">
  <br>
  <button id="render-btn">万単位のノードレンダリングをトリガー</button>
  <div id="status"></div>
  <div id="app"></div>

  <script>
    // Stack Reconciler の再帰レンダリングをシミュレート
    function renderTree(depth) {
      if (depth === 0) {
        const el = document.createElement('div');
        el.className = 'box';
        // CPU を消費するコンポーネントロジックをシミュレート
        let sum = 0;
        for (let i = 0; i < 50000; i++) sum += Math.random();
        return el;
      }
      
      const container = document.createElement('div');
      // 各階層に二つの分岐があり、指数関数的に増加
      container.appendChild(renderTree(depth - 1));
      container.appendChild(renderTree(depth - 1));
      return container;
    }

    document.getElementById('render-btn').addEventListener('click', () => {
      const app = document.getElementById('app');
      const status = document.getElementById('status');
      
      status.textContent = 'Rendering... ページが固まりました。すぐにタイピングして！';
      
      // setTimeout を使い、上のステータス文字が描画されてからメインスレッドを占拠し始めるようにする
      setTimeout(() => {
        const start = performance.now();
        
        // 深さ 13 のツリーを生成。計 8192 個のノード、重い計算付き
        const tree = renderTree(13);
        
        app.innerHTML = '';
        app.appendChild(tree);
        
        const time = (performance.now() - start).toFixed(0);
        status.textContent = `Done in ${time} ms. (さっき打った文字が、今ようやく表示されたでしょう！)`;
      }, 50);
    });
  </script>
</body>
</html>
```
